# tdd-development-generate-test-cases

## 目的
抽出済みユースケースをテストケース表に落とし込み、TDDで着手できる粒度に整える。正常系・異常系・例外系を整理し、テストIDにより追跡可能性を確保する。

## 入力前提
- `/extract-test-targets` の結果、または同等のユースケース情報が会話コンテキストに存在する。
- テスト対象のモジュールやAPIが特定されている。未特定の場合は特定すべき点を質問する。
- 既存のテストID命名規約が分かる場合は従い、なければ `TC-01` 形式で連番を付与する。

## 手順
1. ユースケースをテスト目的ごとに分割し、1テスト=1責務となるよう調整する。
2. 各テストに対して以下を整理する。
   - ID: 命名規則に従う。仕様IDとのマッピングは補足欄に記述する。
   - シナリオ: テストの目的を一行で表現する（正常/異常などを明示）。
   - 前提条件: データベース状態・ロール・FeatureFlagなど。
   - 入力: 具体的なリクエストパラメータ、操作手順。
   - 期待結果: レスポンス/状態変化/副作用。
   - 補足: 関連する仕様ID、境界値、確認事項など。
3. 抜けがある場合は追加テストを提案し、理由を説明する。
4. 仕様が曖昧なままのテストは `未確定` と注記し、確認事項を併記する。

## 出力フォーマット
- Markdown表を使用し、列は `ID`, `シナリオ`, `前提条件`, `入力`, `期待結果`, `補足` を基本とする。
- 表の前にテスト対象コンポーネント名と目的の要約を記述する。
- 表後に、抜けている観点や今後の確認事項があれば箇条書きで示す。

## 応対スタイル
- 丁寧かつプロの視点で、根拠を添えて提案する。
- ユーザーの案がある場合は尊重し、差分や補強点を補足する。
- 不明点は「次に確認したいこと」として明文化し、対話をリードする。
