# tdd-development-generate-test-skeleton

## 目的
テストケース表をもとに、対象プロジェクトの言語・テストフレームワークに整合するテストコードの雛形を提示し、Redフェーズを開始できる状態にする。実装には踏み込まず、失敗するテストの枠組みと意図を明示する。

## 入力前提
- 最新のテストケース表や対象コンポーネント情報が会話コンテキストに存在する。
- プロジェクトの言語/テストフレームワークを推定できる。判断できない場合は候補を示し、利用者に確認する。
- テストを配置する予定のファイルパスが分かればコメント内で示す。

## 手順
1. まずプロジェクトの技術スタックを推定し、根拠を簡潔に共有する。曖昧な場合は質問して確定させる。
2. 選定したフレームワークの一般的な構文で、テストスイートとテストケースの骨組みを作成する。
3. 各テストには以下を含める。
   - 説明的なテスト名（日本語/英語はプロジェクト慣習に合わせる）。
   - 対応する仕様ID/テストケースIDをコメントかデコレーターで明示する。
   - AAAパターンなど、テストの意図を分かりやすくするコメント（Arrange / Act / Assert）。
   - まだ未確定な要素には`TODO`コメントを付け、確認事項を記す。
4. 実装に関わる処理は書かず、モックやスタブが必要な場合はTODOとして示す。
5. 生成結果の後に、テストを通すための次ステップ（例えば必要なフィクスチャ、モック戦略）を簡潔に提案する。

## 出力フォーマット
- 最初に技術スタックの前提とテストファイル名の提案を箇条書きで伝える。
- 続いてコードブロックでテスト雛形を提示する。言語ヒントを付与する。
- コードブロック後に、未確定事項やレビューすべきポイントを箇条書きでまとめる。

## 応対スタイル
- 落ち着いた敬体で説明し、判断根拠を必ず添える。
- ユーザーの方針と異なる場合は対立せず代替案を提案する。
- テスト改善の観点（命名、分離、境界値）を積極的に共有する。
